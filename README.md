# Dosochka
## 1. Пользователи и сессии

Регистрация пользователя по login, display_name и password.
Аутентификация пользователя по login и password.
Создание серверной сессии после успешной регистрации или входа.
Поддержка гостевого участника с nickname без полноценной учетной записи.

## 2. Доски

Создание доски только зарегистрированным пользователем.
Автоматическое назначение создателя владельцем доски (owner).
Хранение у доски названия, режима доступа, политики гостей, пароля при необходимости и текущей ревизии.
Получение актуального snapshot состояния доски.

## 3. Доступ и роли

Поддержка ролей owner, editor, viewer.
Поддержка режимов доступа private, invite_only, password_protected.
Поддержка политик гостей guest_disabled, guest_view_only, guest_edit_allowed.
Определение роли участника при входе на доску на основе типа principal, membership и настроек доски.
Изменение настроек доски только владельцем.
Запрет изменения настроек доски для editor и viewer.

## 4. Модель доски

Представление доски как сцены объектов, а не как изображения.
Поддержка объектов типов stroke, shape, text.
Хранение текущего состояния объектов доски.
Хранение списка активных участников доски.

## 5. Операции над доской

Создание фигуры.
Создание текстового объекта.
Начало stroke.
Добавление точек в stroke.
Завершение stroke.
Изменение геометрии и стиля объектов.
Перемещение объекта.
Изменение размера объекта.
Удаление объекта.

## 6. Правила обработки операций

Передача изменений доски в виде команд.
Фиксация успешно примененных изменений в журнале операций.
Увеличение revision после каждой успешно примененной операции.
Проверка base_revision перед применением команды.
Последовательное применение операций внутри одной доски.
Разрешение на изменение состояния только для owner и editor.
Запрет на изменение состояния для viewer.
Запрет на изменение удаленного объекта.
Запрет на append_stroke_points и end_stroke без ранее начатого stroke.

## 7. Runtime и хранение

Поддержка runtime-состояния активной доски в памяти.
Поддержка runtime-менеджера для последовательной обработки операций по конкретной доске.
Сохранение актуального состояния объектов и истории операций в слой хранения.
Использование in-memory репозиториев для реализации и тестирования бизнес-логики на втором этапе.

Покрытие unit-тестами сценариев регистрации и входа.
Покрытие unit-тестами сценариев создания доски и входа на доску.

Покрытие unit-тестами применения операций и изменения revision.
Покрытие unit-тестами runtime-обработки операций.
